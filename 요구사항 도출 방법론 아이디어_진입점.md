## 방법3

```
요약

- 레거시 코드의 외부 진입점과 효과지점을 출발점으로 삼아 코드 사실을 규칙 후보 형태로 수집합니다.

- 이를 묶고 정리하고 검증하는 과정을 거쳐 최종적으로 요구사항 명세 후보를 도출하는 방법입니다.
```

밖에서 시스템을 건드릴 수 있는 진입점을 먼저 열거하고, 거기서 실제로 발생하는 효과를 따라가며 수집하는 방식입니다.

진입점이 중요한 이유는 시스템이 실제로 어떤 자극을 받아 움직이기 시작하는가를 보여주기 때문입니다.

요구사항은 보통 무언가가 요구되면 시스템은 어떻게 반응해야하는가의 형태인데 진입점은 그 무언가가 들어오는 순간을 잡아줄 수 있다고 판단했습니다. 요구사항을 단번에 잡을 수 있는 것은 아니고 요구사항 후보를 잃지 않고 추적하기 위한 출발점으로 선택하기 적합해보입니다.

#### 1. 진입점을 열거합니다.

밖에서 이 프로그램을 건드릴 수 있는 시작점(entrypoint)들을 전부 찾아서 목록으로 만드는 것입니다.

예를들면 다음이 있을 수 있습니다.

- 사용자가 웹/앱에서 눌러서 들어오는 요청
- 다른 시스템이 API로 호출하는 지점
- 메시지 큐, 이벤트, 웹훅처럼 외부가 밀어 넣는 지점
- 크론, 배치, 스케줄러가 자동 실행하는 작업

이건 의미 해석 뿐만 아니라 프레임워크 규칙으로도 찾을 수 있습니다.

하지만 Go같은 경우는 어노테이션 위주가 아니라 관례와 표준 패턴을 보는 경우가 더 많기 때문에
진입점 후보는 보통 main에서 시작해 서버 부팅 코드, 라우터 등록 코드, 핸들러 연결 코드, 메시지 consumer 등록, 스케줄러/배치 등록, CLI command wiring 코드를 따라가며 찾도록 해야 할 것입니다.

#### 2. 각 진입점에서 효과 지점만 찾습니다.

진입점을 찾은 다음에는 그 진입점이 실제로 시스템에 어떤 눈에 보이는 결과를 남기는지 보자는 것입니다.

시스템이 실제로 의미 있는 일을 했다는건 보통 아래에서 관찰할 수 있습니다.

- DB write
- 상태 필드 변경
- 외부 API호출
- 이벤트 발행
- 파일 생성
- 이메일/알림 전송..

entrypoint의 실제 영향들을 가져오도록 하는 겁니다. 위의 예시는 write 기반으로 되어있지만 실제로는 조회 기능도 충분히 의미 있는 일이라고 생각하기 때문에 직접적으로 영향을 미치지 않더라도 시스템의 중요한 행동을 드러내는 것이라면 다 수집하라고 요청해야합니다.

예를들면 조회 API는 진입점이 되지만 특별히 뭔갈 바꾸는 영향은 주지 않고 read-only의 성격만 가지고 있습니다. 하지만 이는 "어떤 사용자에게는 결과를 보여준다" 이런 행동과 이어질 수 있기 때문에 수집해야합니다.

#### 3. 효과 지점으로부터 슬라이스를 생성합니다.

어떻게 코드가 구성이 되었는지 방법이 중요한게 아니기 때문에 진입점에서 끝까지 전부 읽지 말고 결과가 실제로 나타나는 지점을 잡은 다음 그 결과가 나오기 위해서 꼭 필요한 행동 계약 즉 그 효과를 발생시키는데 필요한 최소 범위를 backword/forward로 닫습니다.

즉 2번에서 잡은 효과 지점이 생기기까지 필요한 코드만 추려서 범위를 닫습니다.

예를 들어 OrderCompleted 이벤트 발행이라는 효과 지점이 있다면

- backword로는
  - 주문 상태가 Approved였는지
  - validation이 통과했는지
  - 어떤 함수 호출을 거쳐 왔는지
- forward로는
  - 이벤트 발행 후 알림이 붙는지
  - 후속 업데이트가 있는지

이렇게 보도록 합니다.

다시 풀어서 설명하면 주문 완료 요청이 들어왔을때 알고싶은건 언제 완료가 허용되는지, 어떤 경우 거부되는지, 완료되는 상태가 어떻게 바뀌는지입니다. 결과가 찍히는 지점부터 거꾸로 추적해서, 그 결과를 만드는 핵심 규칙만 뽑아낼 수 있는 슬라이스를 생성하는 것 입니다.

#### 4. 슬라이스에서 코드 사실을 추출하고 Rule Record 형식으로 정리합니다.

슬라이스를 생성했다고 그 코드를 보고 바로 "승인된 주문만 완료 가능", "완료 시 알림 발송" 이런 자연어 후보를 만들 수 있는게 아닙니다. 그렇게 하면 여기에 코드 해석이 꽤 많이 들어가기 때문입니다.

그래서 이 단계에서는 코드가 직접 보여주는 사실만 추출한 뒤 이를 Rule Record 스키마에 맞춰 정리해야합니다.

예를들어 수강 신청 확정 처리와 관련된 코드가 아래와 같다고 하면

```
enrollment.status != REQUESTED 이면 ErrInvalidState
성공 시 enrollment.status = ENROLLED
enrollment.confirmedAt = now()
PublishEnrollmentConfirmed(enrollment.ID)
```

이 단계에서는 다음과 같은 수준의 코드 사실을 뽑습니다.

```
subject_symbol: enrollment
state_field: enrollment.status
guard_predicates:
  - enrollment.status == REQUESTED
guard_failure:
  - ErrInvalidState
success_writes:
  - field: enrollment.status
    value: ENROLLED
  - field: enrollment.confirmedAt
    value: now()
side_effects:
  - kind: publish
    target: PublishEnrollmentConfirmed
domain_context:
    source_module: service/enrollment.go
    route: POST /enrollments/{id}/confirm
    primary_tables: [enrollments]
    entity_types: [Enrollment]
trigger:
    actor: authenticated_user
    action: confirm_enrollment
```

그리고 이렇게 추출한 사실을 자율서술로 흩어놓지 않고 rule_id, entrypoint_id, evidence, confidence 등의 메타데이터를 포함한 Rule Record로 저장합니다.

> **굳이 고정된 스키마에 맞춰 저장해야하는가**
>
> 코드 사실을 자유서술로 남기면, 같은 규칙이 서로 다른 표현으로 흩어져 나중에 비교 및 병합이 어려워진다.
> 반면 고정된 필드 구조를 사용하면 서로 다른 진입점이나 코드 경로에서 추출된 사실도 같은 기준으로 정렬할 수 있을 것이다.

> **Rule Record의 필드는 어떻게 정해야할까?**
>
> Design by Contract: 소프트웨어 구성요소가 정확히 정의된 계약에 따라 협력하도록 하는 방법이며 특정 계약 요소를 강조한다.
>
> - 전제 조건: 언제 가능해야하는가
> - 사후 조건: 하고 나면 무엇이 보장되어야 하는가
> - 불변성: 항상 깨지면 안되는 조건이 무엇인가
>
> DbC는 코드를 읽을 때 최소한 위의 세 가지를 안정적으로 붙잡게 해 준다.
>
> 요구사항 명세에서 정말 필요한 것은 결국 다음과 같은 정보다
>
> - 어떤 조건에서 기능이 허용되는지
> - 어떤 경우 실패해야 하는지
> - 성공하면 어떤 상태와 변화와 결과가 생기는지
> - 어떤 제약은 항상 유지되어야 하는지
>
> 따라서 DbC의 전제조건, 사후조건, 불변성은 좋은 기준점이 될 것이라 판단하였다.
>
> ---
>
> 그래서 나온 필드가 다음과 같다.
>
> - subject_symbol: 해당 규칙이 주로 걸려 있는 중심 객체 또는 변수이다.
> - state_field: 그 객체에서 상태 전이나 허용 조건 판단에 핵심적으로 사용되는 필드이다.
> - guard_predicates: 동작이 허용되기 전에 만족되어야 하는 조건들이다, 이는 DbC의 전제조건에 대응한다.
> - guard_failure:전제조건이 만족되지 않았을 때 발생하는 실패 경로 또는 오류이다, 실무적으로는 “언제 불가능한가”도 중요하므로 전제조건과 함께 별도 관리한다.
> - success_writes: 동작이 성공했을 때 기록되는 상태 변화, 값 변경, 결과 반영이다, 이는 DbC의 사후조건 중 상태 변화와 결과 보장에 대응한다.
> - side_effects: 성공 후 이어지는 이벤트 발행, 외부 호출, 알림 전송 등 부수효과이다, 이 역시 사후조건의 일부로 볼 수 있으나, 후속 분석을 위해 별도 필드로 둔다.
> - domain_context: 해당 규칙이 어떤 모듈, 라우트, 테이블, 엔티티 타입에서 왔는지를 기록하는 출처 정보이다. evidence가 "어느 코드 위치에서 관찰되었는가"를 기록한다면, domain_context는 "어떤 도메인 경계 안에 속하는가"를 기록한다. 모듈 경로, 라우트명, 테이블명은 코드에서 직접 읽을 수 있는 사실이므로 해석이 아니다.
> - trigger: 이 행동을 촉발하는 주체(actor)와 동작(action)이다. 진입점의 종류와 이름에서 직접 읽을 수 있다. 요구사항의 "~하면 시스템은 ~한다" 구조에서 "~하면" 부분의 원료가 된다.
> - evidence: 해당 사실이 어떤 파일, 함수, 분기, 호출 경로에서 관찰되었는지를 기록하는 근거 정보이다.
> - entrypoint_id: 이 규칙이 어떤 진입점 분석 과정에서 추출되었는지를 추적하기 위한 식별자이다.
> - rule_id: 각 규칙 레코드를 고유하게 식별하기 위한 ID이다.
>
> Rule Record의 필드는 DbC의 전제조건, 사후조건, 불변성이라는 축을 중심으로 하되, 레거시 분석 실무에서 반드시 필요한 실패 경로, 상태 변화, 부수효과, 근거, 추적 정보를 덧붙여 확장한 것이다.

> 불변성에 관련된걸 현 시점에서 파악하기 어렵다는 것이 문제이다. 왜냐하면 불변성은 한 코드 슬라이스에서 바로 읽히기 보다, 여러 경로에 걸쳐 반복되거나 write 이후에도 계속 유지돼야 하는 제약으로 드러나는 경우가 많기 때문이다.

#### 5. Rule Record들을 도메인 우선으로 묶습니다.

앞 단계까지 오면 Rule Record가 여러개 생깁니다.

그런데 레거시에서는 같은 규칙이 한 군데에만 있지 않고 여러 위치에 반복해서 나타나는 경우가 많습니다. 이걸 전부 따로 두면, 사실은 같은 규칙인데도 여러 장의 카드처럼 흩어져 보이는 겁니다.

그래서 이 단계에서는 Rule Record들을 묶어서 규칙 묶음으로 만드는데, 두 단계에 걸쳐 진행합니다.

#### 5-1. 도메인 컨텍스트 기반 1차 그룹핑

먼저 4단계에서 기록한 domain_context의 코드 사실을 기준으로 같은 도메인에 속하는 Rule Record들을 묶습니다.

같은 source_module + 같은 primary_tables → 같은 도메인 그룹

예를 들면 다음과 같이 묶을 수 있습니다.

```
    DG-ENROLLMENT: EP-031, EP-032, EP-045
                   (모두 service/enrollment.go, enrollments 테이블)
    DG-PAYMENT:    EP-051, EP-052, EP-053
                   (모두 service/payment.go, payments 테이블)
    DG-SCHEDULE:   EP-071, EP-072
                   (모두 handler/schedule.go, schedules 테이블)
```

이것은 "enrollment니까 수강 신청이네"라는 의미 해석이 아닙니다. enrollment.go에 있는 것들을 같이 묶는 것은 auth guard가 같은 것들을 같이 묶는 것과 동일한 수준의 사실 기반 분류입니다. 모듈 경로, 라우트 접두사, 테이블명은 코드에서 직접 읽은 사실이며, 이를 그룹핑에 사용하는 것은 추론이 아닙니다.

이렇게 묶는 이유는, 구조만으로 묶으면 도메인이 다른 것들이 합쳐지기 때문입니다. 예를 들어 auth guard → DB insert → response 구조는 수강 신청, 결제, 일정 등록에서 동일하게 나타나지만 이것들이 하나의 클러스터가 되면 어떤 기능 명세도 도출할 수 없습니다.

#### 5-2. 그룹 내 구조적 패턴 식별

도메인 그룹이 만들어지면, 각 그룹 안에서 기존 방식대로 구조가 같은 Rule Record들을 묶습니다.

예를 들어 여러 코드 경로에서 이런 Rule Record가 나왔다고 가정하면

```
  Record A (EP-031, handler)
    - enrollment.status == REQUESTED
    - 성공 시 enrollment.status := ENROLLED
    - 성공 시 enrollment.confirmedAt := now()
    - side effect: PublishEnrollmentConfirmed

  Record B (EP-032, service layer)
    - e.status == REQUESTED
    - 성공 시 e.status := ENROLLED
    - 성공 시 e.confirmedAt := now()
    - side effect: PublishEnrollmentConfirmed
```

변수명은 다르지만 구조는 거의 같습니다. 이 둘은 같은 도메인 그룹(DG-ENROLLMENT) 안에서 하나의 클러스터로 묶을 수 있습니다.

```
  domain_group: DG-ENROLLMENT
  cluster_id: DG-ENROLLMENT-C-001
  shared_structure:
    state_field_role: *.status
    guard:
      - status == REQUESTED
    success_writes:
      - status := ENROLLED
      - confirmedAt := now()
    side_effects:
      - PublishEnrollmentConfirmed
  member_rules:
    - R-101
    - R-208
```

하지만 완전히 동일해야만 묶는 건 아닙니다. A와 B는 완전히 같지만 C는 추가로 side_effect가 하나 더 있을 수 있습니다. 이런 경우에도 core는 같고 C만 변형일 수 있습니다. 따라서 에이전트가 판단할대는 변형과 예외도 고려할 수 있도록 인스트럭션을 넣어줘야합니다.

#### 5-3. 횡단 관심사 분리

도메인 그룹핑 후에도 모든 그룹에 공통으로 나타나는 구조가 있을 수 있습니다. 예를 들어 인증 guard, 입력 검증,
트랜잭션 롤백 같은 패턴은 수강 신청에도, 결제에도, 일정에도 동일하게 존재합니다.

이것들은 특정 도메인의 기능이 아니라 시스템 전체에 걸친 횡단 관심사입니다. 도메인 그룹 안에 남겨두면 모든 그룹에서
중복 기술되므로, 별도의 횡단 관심사 그룹(XC)으로 추출합니다.

- XC-AUTH: 모든 보호된 엔드포인트는 JWT 인증 guard를 통과해야 한다
- XC-TX: 데이터 변경은 DB 트랜잭션 내에서 수행되어야 한다
- XC-VALID: 입력 파라미터는 빈 값, 형식 검증을 통과해야 한다

횡단 관심사는 최종 문서에서 비기능 요구사항 또는 아키텍처 제약 섹션에 배치됩니다.

#### 6. 도메인 그룹에서 기능을 도출하고, 충분한 경우 명세 후보로 승격합니다.

이 단계에서는 앞서 묶은 도메인 그룹을 대상으로, 그 그룹이 사용자에게 제공하는 기능이 무엇인지 도출하고, 그 근거가 충분하다고 판단되는 경우에만 명세 후보로 승격합니다.

이 단계의 질문은 "이 도메인 그룹의 진입점들을 통해 사용자(또는 시스템)가 할 수 있는 것은 무엇인가?"입니다.

질문이 위와 같아야하는 이유는 요구사항 명세의 기본 단위가 사용자가 할 수 있는 기능이기 때문입니다.

기능 도출 절차는 각 도메인 그룹에 대해 다음을 수행합니다.

a) 진입점 목록에서 기능 후보를 열거합니다. 진입점 이름은 코드 사실이므로 이 자체가 기능 후보의 1차 원료가 됩니다. 또한 이미 이 진입점 기준으로 모든 Rule Record들이 형성되어있어서 각 진입점이 무엇을 하고 있는지를 판단할 수 있게 해주는 근거가 됩니다.

DG-ENROLLMENT의 진입점들:

- POST /enrollments → 수강 신청을 생성할 수 있다
- POST /enrollments/{id}/confirm → 수강 신청을 확정할 수 있다
- GET /enrollments → 수강 신청 목록을 조회할 수 있다
- DELETE /enrollments/{id} → 수강 신청을 취소할 수 있다

b) 각 기능 후보에 Rule Record의 행동 정보를 붙입니다.

- 기능 후보: 수강 신청을 확정할 수 있다
  - precondition: enrollment.status == REQUESTED
  - postcondition: enrollment.status := ENROLLED, confirmedAt := now()
  - side_effect: PublishEnrollmentConfirmed
  - guard_failure: status != REQUESTED → ErrInvalidState

c) 이때 grounding을 수행합니다. 기존과 마찬가지로 변수명, 타입명, 라우트명, 테이블명, 이벤트명, 테스트명, UI
문자열, 문서 및 주석을 검토합니다.

도메인 그룹 안에서 수행하므로 이미 같은 도메인의 단서들이 모여 있어 grounding 성공률이 높습니다. 예를 들어
DG-ENROLLMENT 그룹에서는:

- EnrollmentService
- /enrollments/{id}/confirm
- enrollments 테이블
- PublishEnrollmentConfirmed

이 모든 단서가 같은 방향을 가리키므로 자연스럽게 도메인 용어를 부여할 수 있습니다.

중요한 점은 여전히, 무조건 이름을 붙이거나 요구사항 문장으로 올리는 것이 아니라, 붙일 수 있으면 붙이고, 아니면
붙이지 않는 것입니다. 근거가 약하거나, 이름은 있어도 의미가 불분명하거나, 서로 상충하는 단서가 존재한다면
ungrounded 상태로 유지합니다.

이 단계를 거친 결과물:

```
{
  "domain_group": "DG-AUTH-CLIENT",
  "features": [
    {
      "feature_id": "F-AUTH-CLI-001",
      "capability": "사용자가 username과 password로 회원가입할 수 있다",
      "trigger": {
        "actor": "unauthenticated_user",
        "action": "tauri::command register(username, password)"
      },
      "behavior": {
        "preconditions": [...],
        "postconditions": [...],
        "side_effects": [...],
        "guard_failure": [...]
      },
      "evidence_refs": [...],
      "grounding_confidence": "high",
    }
  ]
}
```

#### 7. 승격된 명세 후보를 요구사항 문서 구조에 맞게 재구성합니다.

앞 단계의 도메인 그룹별 기능 후보들은 아직 분석 순서와 그룹 경계에 따라 만들어졌기 때문에 그대로는 이해하기 좋은
요구사항 문서가 되기 어렵습니다.

따라서 이미 승격된 기능 후보를 대상으로, 그것이 문서 수준에서 어떤 의미를 가지는지를 기준으로 다시 배열하고
편집해야 합니다.

이 단계에서 수행하는 작업은 다음과 같아야 합니다.

a) 의미상 동일한 규칙 병합

서로 다른 도메인 그룹 내 클러스터에서 나왔더라도 결과적으로 같은 명세 의미를 가지는 항목은 하나의 요구사항으로
통합합니다.

예를 들어 서로 다른 클러스터가 아래와 같이 승격되었다고 하면

- DG-ENROLLMENT-C-01: 신청 상태에서 등록 확정 성공 시 confirmedAt 기록
- DG-ENROLLMENT-C-07: 관리자 처리 경로에서 등록 확정 성공 시 confirmedAt 기록
- DG-ENROLLMENT-C-13: 배치 처리 경로에서 등록 확정 성공 시 confirmedAt 기록

이들은 코드 경로도 다르고, 일부 예외나 부수효과도 달라서 클러스터 단계에서는 별개의 묶음으로 남아있을 수 있습니다.
하지만 명세 문서 수준에서 보면 셋 모두 결국 하나의 사후조건으로 읽힐 수 있습니다. 이때 클러스터 단위의 근거는
사라지지 않고 evidence_refs 형태로 함께 유지합니다.

b) 예외 결합

기본 규칙과 예외 규칙을 따로 떨어뜨려 두면 문서가 찢어집니다. 따라서 예외는 가능한 한 기본 기능에 연결하여, 독자가
기본 규칙과 예외 관계를 한 번에 이해할 수 있도록 정리합니다.

c) 충돌 표시

서로 다른 클러스터에서 승격된 후보가 상충되는 경우에는 강제로 병합하지 않습니다. 예를 들어

- DG-ENROLLMENT-C-02: 신청 상태일 때만 등록 확정 가능
- DG-ENROLLMENT-C-22: 등록금 납부 상태이면 등록 확정 가능

처럼 상충하는 규칙이 나오면, 이를 하나의 문장으로 합치지 않고 conflict로 표시합니다.

이 단계를 거친 결과물은 다음과 같은 형태가 나오도록 요청합니다.

```
dstructured_spec:
  domains:
    - domain: Enrollment
      features:
        - id: FR-ENROLL-001
          capability: 수강 신청을 확정할 수 있다
          trigger:
            actor: authenticated_user
            action: confirm_enrollment
          behavior:
            preconditions:
              - 신청 상태(REQUESTED)의 수강 신청에 대해서만 허용해야 한다
            postconditions:
              - 상태를 ENROLLED로 변경해야 한다
              - 확정 시각을 기록해야 한다
            side_effects:
              - EnrollmentConfirmed 이벤트를 발행해야 한다
          evidence_refs:
            - DG-ENROLLMENT-C-01
            - DG-ENROLLMENT-C-05
            - DG-ENROLLMENT-C-12
          exceptions:
            - 관리자 일괄처리 경로 예외 가능성 있음

  cross_cutting:
    - id: XC-AUTH-001
      type: architectural_constraint
      statement: 모든 보호된 엔드포인트는 JWT 인증 guard를 통과해야 한다
      applies_to:
        - FR-ENROLL-*
        - FR-PAYMENT-*
        - FR-SCHEDULE-*
      evidence_refs:
        - ...

  conflicts:
    - conflict_id: CF-01
      description: DG-ENROLLMENT-C-02의 preconditions와 DG-ENROLLMENT-C-22가 상충
      refs:
        - DG-ENROLLMENT-C-02
        - DG-ENROLLMENT-C-22

  unresolved:
    - EnrollmentConfirmed가 외부 알림인지 내부 이벤트인지 불명확

```

#### 8. 마지막으로 나온 문서를 기반으로 검증 및 보강 루프를 돌립니다.

지금 만든 요구사항 명세가 실제 코드와 도메인에 비춰 봤을 때 버틸 수 있는지 계속 깨 보고, 약한 부분을 보강하는 단계입니다.

a) 추가 근거 확보

말 그대로 증거를 더 모으도록 합니다.

특히 앞 단계에서 아래와 같은 상태였던 것들의 증거를 더 모읍니다.

- 도메인 용어를 못 붙여서 ungrounded로 남아있는 경우
- system-level spec까지만 올라가 있고 domain-level로 못간 경우
- blocked item으로 보류된 경우

이런 것들에 대해, 이걸 더 뒷받침해 줄 증거가 다른 곳에 없는지 코드베이스로 다시 찾습니다.

b) 반례 탐색

현재 만든 규칙을 깨는 코드 경로나 예외 경로가 있는지 찾습니다.

예를 들어 지금 규칙 후보가

- Approved일 때만 성공

이라고 되어 있어도, 실제로는

- batch 경로에서는 예외 허용
- 관리자 role은 우회 가능
- 특정 API에서는 다른 조건으로 성공

같은 반례가 있을 수 있습니다. 이와 같은 반례를 통해 규칙 강화 및 예외를 붙일 수도 있습니다.

c) 기능 커버리지 검증

모든 진입점(EP)이 최소 하나의 기능 요구사항(FR) 또는 횡단 관심사(XC)에 매핑되었는지 확인합니다.

매핑 안 된 EP가 있으면:

- 해당 EP의 Rule Record를 다시 확인
- 새로운 도메인 그룹을 만들어야 하는지 검토
- 유틸리티/내부 전용 EP인 경우 비기능 요구사항 또는 분석 제외 대상으로 분류

모든 EP가 어딘가에 매핑되지 않으면 검증이 완료되지 않은 것입니다.

d) 문서 참고

코드만으로는 도메인 규칙에 적절한 단어를 못 찾을 수 있습니다.

예를 들어

- 이 상태를 학사적으로 뭐라고 부르는지
- 이 처리의 공식 업무 명칭이 뭔지
- 이 이벤트가 외부 통지인지 내부 후속 처리인지

이런 건 코드만으로는 애매할 수 있습니다.

이럴때 기존에 있던 문서를 활용하도록 합니다. 지금까지 쌓아온 운영 자료들(git commit message, drive 문서)을 참고 근거로 볼 수 있습니다.
