# 요구사항 도출 방법론

레거시 코드를 기능은 그대로 유지하면서 완전히 재작성하기 위해, 코드에서 요구사항 명세를 역으로 추출하는 방법론이다.
코드에서 자연어로 해석하는 것을 최대한 뒤로 미루고, 코드 사실만을 먼저 수집한 뒤 점진적으로 추상화해 나간다.

---

## 왜 이 방법론인가

레거시 코드는 문서가 없거나 낡아있는 경우가 대부분이다. 그렇다고 코드를 처음부터 끝까지 읽는 방식은 규모가 커지면 비효율적이고, LLM에게 임의로 grep하게 두면 출발점이 일관성 없이 잡혀 누락이 발생한다.

이 방법론은 세 가지 원칙을 기반으로 한다.

1. **구조적 출발점** — AST로 파싱된 코드 구조를 기반으로 진입점을 체계적으로 열거한다. grep 기반 탐색보다 누락이 적고 의미 단위가 명확하다.
2. **사실 우선, 해석 나중** — 코드에서 직접 읽히는 사실(guard 조건, 상태 변경, 에러 코드)만 먼저 수집하고, 의미 해석은 도메인 그룹이 만들어진 이후에만 수행한다.
3. **근거 추적 가능성** — 모든 요구사항은 어느 코드에서 왔는지 `evidence_refs`로 연결된다. 요구사항 문서가 완성된 후 "이 코드는 어느 요구사항에 반영됐는가"를 역방향으로 검증할 수 있다.

---

## 전체 파이프라인

```
[Phase 1] extract    → 백엔드 AST + Call Graph + 클라이언트 정적 분석
[Phase 2] slice      → 진입점 열거 → 효과지점 탐색 → Rule Record 추출 + 스키마 보강 + 클라이언트 컴포넌트 슬라이싱 (CL Rule Record)
[Phase 3] group      → 도메인 그룹핑 → 구조 클러스터링 → 횡단 관심사 분리 (교차검증)
[Phase 4] field      → 필드 생애주기 분석 → 불변 제약 · 조건부 필수 도출
[Phase 5] derive     → 기능 도출 → 명세 후보 승격 → 요구사항 문서 재구성 + 스키마 전파 (교차검증)
[Phase 6] audit      → Reverse Audit → 커버리지 검증 → UI 커버리지 역검증
[Phase 7] reconcile  → enum 진실표 수집 → 문서 스캔 → 불일치 수정
```

중간 산출물은 단계별로 저장된다. 각 단계는 이전 단계의 파일을 읽어 다음 단계의 파일을 생성하므로 중간 재실행이 가능하다.

**산출물 구조 (예시)**

```
analysis_output/              — 백엔드 AST (파일별 JSON)
call_graph.json               — 내부 함수 간 Call Graph
client_output/
├── api_map.json              — 클라이언트 API 호출 메서드 → 엔드포인트 매핑
├── routes.json               — 라우트 + 권한 분기
├── component_map.json        — 컴포넌트 → API 호출 체인
├── client_behaviors.json     — 클라이언트 행동 패턴 (폴링, 조건 분기, 확인 가드)
├── display_labels.json       — 상태 코드 → 표시 텍스트 매핑
└── ui_structure.json         — 페이지별 UI 구조 (컬럼, 폼 필드 등)

.req-analysis/
├── 1_entry_points.json       — 진입점 목록
├── 2_effect_points.json      — 효과 지점 목록
├── 3_rule_records.json       — Rule Record (request/response 스키마 포함)
├── 4_domain_groups.json      — 도메인 그룹 + 클러스터
├── 5_field_spec.json         — 필드 생애주기 제약 (불변 · 조건부 필수 등)
├── 6_features/
│   └── DG-{ID}.json          — 도메인별 기능 후보
├── 7_structured_spec.json    — 최종 기능 요구사항 (FR + XC)
├── 8_audit_report.json       — 커버리지 검증 결과
├── 9_reconcile_report.json   — enum 불일치 수정 내역
└── UI_requirements.md        — 화면 구성 요구사항 (How It Looks)
```

## Phase 1 — extract

### 핵심 개념

AST(Abstract Syntax Tree)는 코드를 문법 구조로 변환한다. grep과 달리 함수 경계, 호출 관계, 타입 정보를 정확하게 파악할 수 있다.

백엔드는 언어에 맞는 AST 파서(예: Go → tree-sitter-go, Java → javaparser)로 파싱한다. 클라이언트 코드는 언어와 프레임워크에 따라 정적 파싱 수준을 결정한다. 구조가 예측 가능한 프레임워크(예: Vue 2 Options API)는 정규식으로도 충분히 커버된다.

### 백엔드 AST 산출물

파일별 JSON에 다음을 포함한다:

- `structs` — struct/class 이름, 필드(이름, 타입, 직렬화 태그)
- `functions` — 함수명, 시그니처, 호출 목록, DB 접근, 에러 메시지
- `db_access` — DB 쿼리 실행 지점 + SQL 힌트

### 클라이언트 정적 분석

클라이언트 분석은 신뢰도 수준에 따라 세 가지로 나뉜다:

**Level 1 — 100% 정적 사실 (hallucination 없음)**
- API 클라이언트 메서드 → HTTP method + URL 매핑
- 라우터 권한 설정 → 역할 목록 추출 (비트마스크, 배열 등 형식에 따라 디코딩)
- 컴포넌트 → API 호출, 상태 관리 action 호출, lifecycle 호출 체인
- 컴포넌트 → 라우터에서 추출한 `route_paths` 매핑
- **상태 코드 → 표시 텍스트 매핑**: 상태값 변환 모듈(필터, 파이프, 유틸 함수 등)의 switch/map 파싱 + UI template의 select option 패턴 → `display_labels.json`
- **페이지 UI 구조**: 페이지 제목, 테이블 컬럼 헤더, 폼 필드 레이블-입력 쌍, 반복 렌더링 데이터 필드 → `ui_structure.json`
- **Validation 메시지 + API 파라미터 쌍**: `$swal("메시지")` 호출에서 텍스트와 트리거 조건을 추출하고, 같은 컴포넌트의 API 파라미터 이름 목록과 함께 기록 → `component_map.json`의 `validation_messages` + `api_params`

**Level 2 — 정적 단서 + AI 연결 (낮은 hallucination)**
- 폴링 패턴 → 간격, 종료 조건 구조 추출
- 조건부 API 호출 패턴 → 분기 조건 구조 추출
- 불가역 작업 확인 패턴 → 확인 텍스트 추출
- 버튼/입력 비활성화 조건 → 조건 추출
- **`string` 타입 파라미터의 실제 의미 판단**: 백엔드 타입이 `string`이지만 UI가 선택 위젯(버튼 그룹, 체크박스 등)으로 채우는 경우, validation 메시지와 API 파라미터 쌍을 AI에게 제공해 연결을 판단하게 한다. AI는 코드 인용으로 근거를 제시해야 하며, 인용 불가 시 `confidence: "low"` 처리.

**Level 3 — 순수 AI (사용 안 함)**

UI 레이아웃 의미 해석, 컴포넌트 목적 추론 → 이 수준은 적용하지 않는다.

### 두 가지 요구사항 문서의 기원

Phase 1에서 추출하는 클라이언트 정보는 최종적으로 두 종류의 요구사항 문서로 분리된다.

- `6_structured_spec.json` — **기능 요구사항** (What it does): 백엔드 코드에서 도출, LLM이 해석
- `UI_requirements.md` — **화면 구성 요구사항** (How it looks): 클라이언트 코드에서 정적 추출, LLM 개입 최소

두 문서를 분리하는 이유는 신뢰도가 다르기 때문이다. 클라이언트 코드에서 정적으로 추출한 UI 문서는 hallucination이 없지만, LLM이 코드 사실을 자연어로 변환한 기능 요구사항에는 hallucination이 섞일 수 있다. Phase 6 reconcile에서 이 비대칭을 이용해 불일치를 자동으로 수정한다.

### 클라이언트 정보의 역할

백엔드가 시스템이 **무엇을** 하는지의 권위라면, 클라이언트는 **누가** **어떤 맥락에서** 하는지의 권위다.

| 클라이언트 산출물 | 파이프라인 소비 위치 | 역할 |
|---|---|---|
| `routes.json` (권한 설정) | Phase 2 slice | Rule Record `trigger.actor` 확정 |
| `display_labels.json` | Phase 4 derive | FR의 `ui.display_labels` 섹션 |
| `client_behaviors.json` | Phase 4 derive | FR의 `ui` 섹션 (display_fields, validation_messages 등) |
| `component_map.json`의 `validation_messages` + `api_params` | Phase 2 slice | `string` 타입 파라미터의 실제 의미 판단 (AI 연결 단계) |
| 순수 클라이언트 기능 | Phase 4 derive | 백엔드 EP 없는 경우만 별도 CL-* FR로 승격 |

### Call Graph 역할

Call Graph는 Phase 2의 슬라이스 생성에서 효과 지점으로부터 진입점까지 backward traversal을 할 때 탐색 경로로 사용된다. 요구사항 문서의 출력물이 아니라 슬라이스 생성의 인프라다.

Call Graph 생성 시 내부 함수만 남기는 필터가 필수다. stdlib, vendor/third-party 패키지 호출은 그래프에서 제거한다.

## Phase 2 — slice

### 핵심 개념

진입점은 시스템이 외부 자극을 받아 움직이기 시작하는 지점이다. 요구사항의 "X가 일어나면 시스템은 Y를 한다" 구조에서 X를 가장 정확하게 잡아주기 때문에 출발점으로 선택한다.

진입점을 찾은 뒤에는 그 진입점이 실제로 어떤 효과를 만드는지 추적한다. 효과 지점이 Y에 해당한다.

### 진입점 유형

코드베이스에 따라 진입점의 형태가 다르다:

- **HTTP 핸들러** — 라우터 등록 패턴(`GET /path`, `POST /path` 등)을 따라가 핸들러 함수를 식별
- **이벤트/메시지 소비자** — 메시지 큐, 이벤트 버스의 구독/리스너 등록 지점
- **스케줄러** — cron 표현식, `@Scheduled`, `time.AfterFunc` 등 시간 기반 트리거
- **CLI 명령** — 커맨드 핸들러 등록 지점

### 슬라이스 단위

**하나의 원자적 인과 연쇄 = 하나의 슬라이스**

진입점 단위로 슬라이스를 만들면 하나의 슬라이스에 여러 규칙이 섞인다. 효과 지점 단위로 만들되, 같은 트랜잭션 안에서 원자적으로 묶인 write들은 하나의 슬라이스로 취급한다. 트랜잭션 밖의 side effect(이벤트 발행, 외부 API 호출)는 별도 슬라이스다.

```
POST /orders/{id}/confirm
  ├── [슬라이스 A] DB write: status = CONFIRMED + confirmedAt = now()  (트랜잭션 내, 하나)
  └── [슬라이스 B] event publish: OrderConfirmed                       (트랜잭션 외, 별도)
```

슬라이스가 작을수록 LLM이 Rule Record를 더 정확하게 추출한다.

### trigger.actor 결정

백엔드 핸들러만 봐서는 누가 호출하는지 알 수 없다. 라우터의 권한 설정이 actor의 유일한 구조적 근거다. 클라이언트 `routes.json`에서 추출한 권한 정보를 참조해 Rule Record의 `trigger.actor`를 결정한다.

```
required_roles: ["Admin"]     → actor: "admin_user"
required_roles: ["Customer"]  → actor: "customer_user"
requires_auth: true (역할 없음) → actor: "authenticated_user"
(없음)                          → actor: "unauthenticated_user"
```

### 에러 코드 처리

에러 코드는 가장 정직한 비즈니스 규칙이다. `ErrInvalidState`, `"order limit exceeded"` 같은 값은 개발자가 어떤 상황을 막으려 했는지를 직접 서술한다.

Rule Record의 `guard_failure` 필드에는 에러 문자열만 저장하지 않는다. 반드시 4-tuple로 저장한다.

```json
"guard_failure": [
  {
    "guard_condition": "order.status != PENDING",
    "error_name": "ErrInvalidState",
    "error_message": "order is not in pending state",
    "http_status": 400
  }
]
```

이 네 값이 묶여야 "어떤 조건에서 → 어떤 이름의 에러가 → 어떤 HTTP 코드로" 실패하는지의 완전한 실패 경로가 된다.

### 스키마 보강 (slice 완료 후)

백엔드 AST에서 추출한 struct 레지스트리를 이용해 각 Rule Record에 `request_schema` / `response_schema`를 추가한다.

핸들러명과 구조체명 사이의 네이밍 컨벤션은 코드베이스마다 다르다. 분석 대상 프로젝트의 컨벤션을 먼저 파악한 뒤 매핑 규칙을 적용한다 (예: `{HandlerName}Param` / `{HandlerName}Response`, `{HandlerName}Request` / `{HandlerName}Result` 등).

구조체는 재귀적으로 전개한다: 포인터 타입, optional 필드, 익명 임베딩, 타입 별칭을 모두 처리한다.

### Rule Record 스키마

```json
{
  "rule_id": "EP-031-R-001",
  "entrypoint_id": "EP-031",
  "subject_symbol": "order",
  "state_field": "order.status",
  "guard_predicates": ["order.status == PENDING"],
  "guard_failure": [
    {
      "guard_condition": "order.status != PENDING",
      "error_name": "ErrInvalidState",
      "error_message": "order is not in pending state",
      "http_status": 400
    }
  ],
  "success_writes": [
    { "field": "order.status", "value": "CONFIRMED" },
    { "field": "order.confirmedAt", "value": "now()" }
  ],
  "side_effects": [
    { "kind": "publish", "target": "OrderConfirmed" }
  ],
  "domain_context": {
    "source_module": "service/order.go",
    "route": "POST /orders/{id}/confirm",
    "primary_tables": ["orders"],
    "entity_types": ["Order"]
  },
  "trigger": {
    "actor": "authenticated_user",
    "action": "confirm_order"
  },
  "evidence": [
    { "file": "service/order.go", "symbol": "ConfirmOrder" }
  ],
  "request_schema": { "struct": "confirmOrderParam", "fields": [] },
  "response_schema": null,
  "confidence": "high",
  "terminology_status": "ungrounded"
}
```

**confidence 기준**
- `high` — 코드에서 직접 읽힌 사실
- `medium` — 한 단계 호출 체인을 추론한 것
- `low` — 이름이나 부분 근거로 추측한 것

### 클라이언트 컴포넌트 슬라이싱

백엔드 EP 분석만으로는 포착되지 않는 요구사항이 있다. 대표적인 경우: **백엔드가 전체 목록을 반환하고 클라이언트가 필터링·정렬을 로컬에서 처리**하는 패턴이다.

이 경우 백엔드 Rule Record에는 "목록을 반환한다"만 기록된다. "졸업학기·배움과정·이름으로 필터링할 수 있다"는 요구사항은 백엔드 슬라이스에 나타나지 않는다.

**접근 방식**: LLM이 클라이언트 컴포넌트 파일을 직접 읽어 판단한다. 정규식 패턴 매칭을 사용하지 않는다. 정규식 기반 추출은 프레임워크·컨벤션에 종속되어 다른 프로젝트에서 재사용할 수 없기 때문이다.

**대상 컴포넌트 선정**: 라우트 경로가 있는 페이지 단위 컴포넌트 중 백엔드 API를 하나 이상 호출하는 것.

**추출 판단 기준**:
1. computed/watch 등에서 백엔드 응답 데이터를 클라이언트가 로컬로 필터링·정렬하는 로직이 있는가
2. 해당 필터 파라미터가 백엔드 EP의 `request_schema`에 없는가 (진짜 클라이언트 전용인가)
3. 사용자 관점에서 의미 있는 데이터 탐색 기능인가 (단순 UX 토글은 제외)

세 조건을 모두 만족하면 **CL Rule Record**로 추출한다.

```json
{
  "rule_id": "CP-001-R-001",
  "source": "client",
  "component": "views/graduate/GraduateCandidate.vue",
  "capability": "졸업 후보를 졸업학기·배움과정·국문이름으로 필터링할 수 있다",
  "behavior_type": "client_filter",
  "filter_fields": [
    { "field": "student.graduated.semester", "label": "졸업학기" },
    { "field": "student.grade",              "label": "배움과정" },
    { "field": "name.korean",                "label": "국문이름" }
  ],
  "reset_action": "searchInit",
  "related_backend_ep": "EP-045",
  "evidence": {
    "file": "views/graduate/GraduateCandidate.vue",
    "symbol": "filteredList",
    "reason": "백엔드 EP request_schema에 필터 파라미터 없음. 클라이언트가 전체 목록 수신 후 로컬 필터링"
  },
  "confidence": "high",
  "terminology_status": "ungrounded"
}
```

CL Rule Record는 백엔드 Rule Record와 함께 `3_rule_records.json`에 저장한다. `source: "client"` 필드로 구분한다.

## Phase 3 — group

### 핵심 개념

같은 규칙이 레거시 코드에서 여러 위치에 반복되는 경우가 많다. 이를 그대로 두면 같은 요구사항이 여러 장의 카드로 흩어진다. 그룹핑은 이 중복을 제거하고 요구사항 후보의 단위를 만든다.

**도메인 기반 1차 그룹핑이 구조 기반보다 먼저 와야 하는 이유**: 구조만으로 먼저 묶으면 주문, 결제, 재고 관리가 같은 클러스터가 된다. 이 세 가지 모두 `auth guard → DB insert → response` 구조를 공유하지만 서로 다른 도메인이다. 도메인 경계를 먼저 그은 뒤 구조 패턴을 찾아야 한다.

### 그룹핑 순서

**0단계 — 클라이언트 라우터 기반 도메인 분리 신호 수집** *(클라이언트 분석 결과가 있을 때만)*

`component_map.json`에서 같은 API/action을 호출하는 컴포넌트들이 서로 다른 라우트 네임스페이스에 속하는지 확인한다. 이것이 "백엔드 기준으로는 하나의 모듈이지만 실제로는 별도 사용자 도메인"인 경우를 미리 포착하는 신호다.

```
규정집.vue   route: /regulation  dispatches: [getBoardList]
공지사항.vue  route: /notice      dispatches: [getBoardList]
학사일정.vue  route: /schedule    dispatches: [getBoardList]
→ 신호: /regulation, /notice, /schedule → DG-BOARD를 3개 도메인으로 분리 후보
```

**라우트 네임스페이스 vs 역할 prefix 구분 기준:**
- 첫 번째 경로 세그먼트가 다른 것(`/orders`, `/payments`) → **도메인 분리** 신호
- 역할 prefix가 다른 것(`/admin/...`, `/customer/...`) → **actor 분리** (`trigger.actor`로 처리, 도메인 분리 아님)
- 같은 도메인 내 CRUD(`/orders`, `/orders/new`, `/orders/:id/edit`) → 하나의 네임스페이스

**1단계 — 도메인 컨텍스트 기반 1차 그룹핑**

`domain_context.source_module`과 `domain_context.primary_tables`가 겹치는 Rule Record들을 같은 도메인 그룹으로 묶는다. 이것은 의미 해석이 아니라 코드 사실 분류다.

0단계에서 클라이언트 라우터 분리 신호가 있으면, 백엔드 기준으로 하나의 그룹이 된 Rule Record들이 여러 라우트 네임스페이스에서 호출된다면 네임스페이스 수만큼 그룹을 분리한다.

**2단계 — 그룹 내 구조 클러스터링**

도메인 그룹 안에서 `state_field_role + guard_predicates + success_writes` 패턴이 같은 Rule Record들을 클러스터로 묶는다. core가 같고 side effect가 하나 더 있는 것은 core cluster + variant로 처리한다.

**3단계 — 횡단 관심사 분리**

모든 도메인 그룹에 공통으로 나타나는 구조(인증 guard, 트랜잭션 wrap, 입력 검증, 권한 확인)는 XC(Cross-Cutting) 그룹으로 분리해 비기능 요구사항 또는 아키텍처 제약으로 분류한다.

### 교차검증 적용 이유

이 단계는 LLM의 추론이 가장 많이 개입되는 구간이다. 클러스터가 잘못 만들어지면 이후 derive 단계가 전부 그 위에 쌓이므로 오류가 증폭된다. 두 모델이 독립적으로 그룹핑을 수행해 결과가 불일치하는 항목을 conflict로 표시한다.

## Phase 4 — field

### 핵심 개념

Phase 2 (slice)는 EP를 독립 단위로 처리한다. 하나의 Rule Record는 "이 EP가 무엇을 하는가"만 담는다.

그런데 다음 요구사항들은 EP를 가로질러 비교해야만 나온다:

- **불변 필드 제약** — "X 필드는 생성 후 수정할 수 없다"
  → 수정 폼에서 해당 input이 disabled 상태. 단일 EP 분석으로는 안 잡힘.
- **조건부 필수 차이** — "Y 필드는 등록 시 필수, 수정 시 선택"
  → 생성 EP와 수정 EP의 request_schema를 비교해야 판단 가능.
- **서버 할당 필드** — "Z 필드는 클라이언트가 전송하지 않는다"
  → response_schema에만 존재하고 어떤 request_schema에도 없는 필드.

Phase 3 (group)이 "어떤 EP들이 같은 도메인인가"를 확정한 뒤에 비로소 이 비교가 가능하다. EP 단위 분석을 아무리 정교하게 해도 이 계층은 구조적으로 안 잡힌다.

### 분석 절차

**Step 1 — EP 유형 분류**

각 도메인 그룹의 EP를 CRUD 유형으로 분류한다.

```
CREATE  — POST + 엔티티 경로
READ    — GET + 엔티티 경로
UPDATE  — PUT / PATCH + 엔티티 경로
DELETE  — DELETE + 엔티티 경로
```

**Step 2 — 엔티티 필드 목록 수집**

도메인 그룹의 모든 EP에서 request_schema와 response_schema를 수집한다.
전체 필드 후보 집합 = 모든 EP의 request + response 필드의 합집합.

각 필드에 대해 EP별 존재 여부와 required 여부를 표로 정리한다:

```
필드명       | CREATE req | UPDATE req | response
------------|-----------|-----------|----------
entity_code | required  | present   | present
name        | required  | required  | present
description | optional  | optional  | present
id          | absent    | absent    | present
```

**Step 3 — UI 정보 교차 참조**

`client_output/ui_structure.json`에서 해당 도메인의 편집 폼 컴포넌트를 찾는다.
편집 폼 식별 기준: 파일명 또는 라우트에 `edit`, `update`, `modify` 등이 포함되거나 라우트에 파라미터(`:id`)가 있는 폼.

편집 폼에서 `immutable: true`로 표시된 필드 → 불변 제약 후보.

**Step 4 — 필드별 생애주기 제약 결정**

| 제약 유형 | 판단 기준 |
|---|---|
| `server_assigned` | 모든 request에 absent, response에만 present |
| `immutable_after_create` | CREATE request에 present, 편집 폼에서 disabled |
| `create_only` | CREATE request에 present, UPDATE request에 absent |
| `always_required` | CREATE와 UPDATE 모두 required |
| `create_required_update_optional` | CREATE required, UPDATE optional |
| `always_optional` | CREATE와 UPDATE 모두 optional |

UI의 `immutable: true` 신호가 스키마 분석보다 우선한다.

### 산출물

`.req-analysis/5_field_spec.json`

```json
{
  "domains": [
    {
      "domain": "Subject",
      "entity": "Subject",
      "fields": [
        {
          "field": "subject_code",
          "constraint": "immutable_after_create",
          "lifecycle": {
            "create": { "present": true, "required": true },
            "update": { "present": true, "required": false, "mutable": false },
            "read":   { "present": true }
          },
          "evidence": {
            "ui": "수정 폼에서 disabled",
            "schema": "updateParam에 포함되나 disabled 처리"
          },
          "derived_requirement": "과목코드는 등록 후 수정할 수 없다."
        }
      ]
    }
  ]
}
```

`derived_requirement`가 null이 아닌 필드는 Phase 5 (derive)에서 해당 FR의 `behavior.preconditions`에 자동으로 포함된다.

### derive 단계 연동

Phase 5 (derive)에서 FR 생성 시 `5_field_spec.json`을 읽어 다음을 자동 포함한다:

| 제약 유형 | FR 반영 위치 |
|---|---|
| `immutable_after_create` | `behavior.preconditions`에 "X는 변경할 수 없다" |
| `create_required_update_optional` | `request_schema.constraints`에 필드 조건 명시 |
| `always_optional` | request_schema 필드 설명에 "(선택)" 표기 |
| `server_assigned` | request_schema에서 제외, response_schema에만 유지 |

## Phase 5 — derive

### 핵심 개념

도메인 그룹에서 "이 그룹의 진입점들을 통해 사용자가 할 수 있는 것은 무엇인가"를 도출하는 단계다. 여기서 처음으로 자연어 기능 명세가 만들어진다.

### 기능 도출 절차

1. 진입점 이름에서 기능 후보를 열거한다 (코드 사실)
2. 각 후보에 Rule Record의 guard/write/side_effect를 붙인다
3. 도메인 용어 grounding을 시도한다 — 코드에서 인용 가능한 근거가 있어야만 용어를 부여한다
4. 근거가 부족하면 `ungrounded` 상태로 유지하고 `blocked_items`에 기록한다

**grounding 가능한 근거 소스**: 변수명, 타입명, 함수명, 라우트명, 테이블명, 이벤트명, 테스트 함수명, 코드 주석, UI 표시 텍스트, UI 조건부 렌더링 조건

**절대 하지 말아야 할 것**: 코드에 없는 용어를 임의로 만들어 붙이는 것.

### 클라이언트 행동 정보 → FR `ui` 섹션 통합

Phase 1에서 추출한 클라이언트 행동 패턴을 별도 CL-* 요구사항으로 분리하는 것이 아니라, 연관된 백엔드 FR의 `ui` 섹션에 결합한다.

**통합 원칙**: 백엔드 EP와 1:1로 대응되는 화면 동작은 해당 FR의 `ui` 섹션에 넣는다. 백엔드 EP가 없는 순수 클라이언트 기능(라우트 가드, 클라이언트 측 필터링)만 CL-* FR로 분리한다.

```
user_messages (입력 검증 조건 + 오류 메시지)  → ui.validation_messages
disabled_conditions (비활성화 조건)           → ui.disabled_conditions
polling (반복 조회 패턴)                      → ui.polling
confirmation_guard (불가역 작업 확인)         → ui.confirmation_guard
display_labels.json (상태 코드 → 표시 텍스트) → ui.display_labels
response_schema.fields                       → ui.display_fields (화면에 표시되는 필드)
```

`display_fields`는 response_schema의 필드 중 실제로 화면에 표시되는 것만 선별한다. 백엔드가 반환하지만 화면에 안 나오는 필드(내부 ID 등)는 제외한다.

**`ui` 섹션 예시:**
```json
"ui": {
  "display_fields": ["year", "semester", "status"],
  "display_labels": {
    "status": {
      "PENDING":   "대기중",
      "CONFIRMED": "확정",
      "CANCELLED": "취소"
    }
  },
  "validation_messages": [
    { "condition": "!title", "message": "제목을 입력해주세요." }
  ],
  "disabled_conditions": [
    { "condition": "!isAdmin || order.status === 'CONFIRMED'", "target": "상태 변경 버튼" }
  ],
  "confirmation_guard": {
    "required_text": "삭제하겠습니다.",
    "context": "불가역 삭제 시 텍스트 입력으로 의도 확인"
  },
  "polling": null
}
```

### 클라이언트 전용 FR 생성 (CL-*)

백엔드 EP가 없는 순수 클라이언트 전용 기능만 별도 CL-* FR로 승격한다.

| 카테고리 | 설명 | 예시 |
|---|---|---|
| `route_guard` | 라우트 접근 권한 제어 | 특정 역할만 접근 가능한 페이지 |
| `client_filter` | 클라이언트 측 데이터 필터링 | 비활성 항목 숨김 처리 |
| `pagination_ux` | 페이지네이션 경계 UX | "첫번째 페이지입니다." 알림 |

`irreversibility_guard`(불가역 확인)와 `async_polling`(비동기 상태 반복 조회)은 대응하는 백엔드 FR이 있으므로 해당 FR의 `ui` 섹션으로 통합한다. 별도 CL-* FR을 만들지 않는다.

### 스키마 전파 (derive 완료 후)

FR의 `evidence_refs` 체인(FR → EP → Rule Record)을 통해 Rule Record의 `request_schema` / `response_schema`를 FR에 복사한다.

### 요구사항 문서 재구성 원칙

- **의미상 동일한 규칙 병합**: 서로 다른 클러스터가 같은 postcondition을 표현하면 하나의 FR로 통합한다. evidence_refs는 모두 유지한다.
- **예외 결합**: 기본 규칙과 예외 규칙을 분리하지 않는다. 독자가 기본 규칙과 예외를 한 번에 볼 수 있게 결합한다.
- **충돌 표시**: 상충하는 규칙은 강제로 병합하지 않는다. `conflict`로 표시해 사람이 판단하게 한다.

### 교차검증 적용 이유

코드 사실에서 자연어 요구사항으로 넘어가는 이 단계가 hallucination이 가장 많이 발생하는 지점이다. 두 모델의 결과가 불일치하면 낮은 confidence 또는 conflict로 표시한다.

## Phase 6 — audit

### 핵심 개념

Reverse Audit은 요구사항 문서가 전체 코드베이스를 얼마나 커버하는지를 역방향으로 검증한다. "요구사항에 한 번도 반영되지 않은 코드"를 찾는다.

이 단계에서 백엔드 커버리지와 UI 커버리지를 함께 검증한다.

### 고아 탐색 전처리

분석 대상 함수 목록에는 유틸 함수, 내부 헬퍼, 초기화 코드가 포함된다. Audit 전에 Call Graph를 이용해 "진입점에서 도달 가능한 함수"와 "도달 불가능한 함수"를 먼저 분리한다. 도달 불가능한 함수는 고아 후보가 아니라 dead code로 별도 분류한다.

**정적 Call Graph의 한계**: 언어에 따라 interface dispatch, reflection, 동적 디스패치 등이 정적 BFS로 추적되지 않는다. 이로 인해 실제로는 호출되지만 "unreachable"로 분류되는 함수가 생긴다. 파일 경로나 패키지 구조를 기준으로 구현 레이어(`database/`, `repository/`, `infrastructure/` 등)를 자동으로 `utility`로 분류하고, BFS 결과만으로 dead code를 단정하지 않는다.

### 백엔드 커버리지 검증

모든 진입점(EP)이 최소 하나의 기능 요구사항(FR) 또는 횡단 관심사(XC)에 매핑되었는지 확인한다.

고아 함수 분류 기준:
- `missed_slice` — Rule Record에는 없지만 핸들러에서 직접 호출됨 → slice 재실행 필요
- `utility` — 여러 도메인에서 공유하는 유틸 함수 → 비기능 요구사항 또는 분석 제외
- `cross_cutting` — 인증 guard, 트랜잭션 래퍼 → XC로 이미 커버
- `truly_orphan` — 위 범주에 해당하지 않음 → 요구사항 누락 가능성

### UI 커버리지 역검증

클라이언트 `component_map.json`을 읽어 백엔드 커버리지와 교차 검증한다.

**방향 1 — FR → UI**: 각 FR의 `ui` 섹션이 있는지 확인
- `ui` 없음 + 내부 API / 배치 작업 → 정상, `internal_api`로 분류
- `ui` 없음 + 일반 기능 → 화면이 누락됐거나 derive 단계에서 UI 매핑이 빠진 것

**방향 2 — UI → FR**: component_map에서 각 컴포넌트가 호출하는 EP 중 어떤 FR에도 없는 것
- derive 단계 누락 → derive 재실행 권고

## Phase 7 — reconcile

### 핵심 개념

Phase 4 (derive)에서 LLM이 자연어 요구사항을 생성할 때 코드값(enum 상수)을 hallucination한다. 가장 흔한 패턴은 두 가지다.

1. **존재하지 않는 값 생성** — 실제 코드 값(`PROCEEDING`)을 그럴듯한 추측값(`ONGOING`)으로 만들어냄
2. **도메인 혼입** — 다른 도메인의 코드값이 엉뚱한 enum description에 섞임 (예: `StatusA`의 마지막 값이 `StatusB`의 값으로 대체됨)

Phase 5 (audit)는 함수 커버리지만 검증하고 **값의 정확성**은 보지 않으므로 이런 불일치가 통과된다. 재구현 AI가 이 값을 그대로 사용하면 런타임 오류가 발생한다.

### 진실표 우선순위

두 요구사항 문서가 다른 값을 가질 때, 아래 순서로 어느 문서를 수정할지 결정한다.

```
1순위  백엔드 타입 정의 파일   — 코드 enum 상수. 최우선 진실표.
2순위  UI 요구사항 문서        — 상태값 변환 모듈에서 정적 추출. LLM 개입 없으므로 신뢰도 높음.
3순위  기능 요구사항 문서      — LLM 생성. hallucination 가능성 가장 높음. 주 수정 대상.
```

2순위를 UI 요구사항 문서로 두는 이유: 이 문서는 LLM 추론 없이 코드에서 직접 파싱한 값이므로 LLM이 생성한 기능 요구사항보다 신뢰도가 높다. 오히려 기능 요구사항의 오류를 역으로 탐지하는 기준으로 사용할 수 있다.

### Step 1 — canonical enum 수집

백엔드 타입 정의 파일에서 enum(타입 별칭 + 상수 목록)을 파싱해 도메인별 허용값 목록을 빌드한다. 이것이 모든 문서의 최우선 진실표다.

```
EntityStatus: [PENDING, CONFIRMED, CANCELLED]
UserType:     [ADMIN, CUSTOMER, GUEST]
```

### Step 2 — 문서 스캔

UI 요구사항 문서 → 기능 요구사항 문서 순서로 스캔한다. UI 문서를 먼저 확인해 백엔드 코드와의 불일치 여부를 파악한 뒤, 기능 요구사항 스캔에서 참조한다.

**탐지 대상:**
- 어떤 도메인에도 속하지 않는 대문자 토큰 (존재하지 않는 enum 값)
- 올바른 값이지만 다른 도메인 enum에 속하는 값이 섞인 경우 (도메인 혼입)
- 상태 전이 문장에 나타나는 잘못된 코드값

**스캔 제외 토큰:** 파이프라인 내부 ID (`FR-*`, `EP-*`, `XC-*`, `CL-*`, `DG-*`), 4자 미만 토큰, 일반 영어 단어 (`NULL`, `TRUE` 등)

### Step 3 — 수정 대상 결정 및 수정

```
불일치 발견
    ├─ 백엔드 코드 ≠ 기능 요구사항 문서     → 기능 요구사항 문서 수정
    ├─ 백엔드 코드 ≠ UI 요구사항 문서      → UI 요구사항 문서 수정
    ├─ 두 문서 불일치, 백엔드 코드는 한쪽과 일치
    │       → 백엔드 코드와 다른 쪽 문서 수정 (통상 기능 요구사항 문서)
    └─ UI 요구사항 문서와 백엔드 코드 모두 다른 값
            → 자동 수정 불가. conflict로 표시.
              상태값 변환 모듈 자체가 백엔드와 drift된 가능성 → 사람이 판단
```

대체값이 명확한 경우 정규식 치환으로 일괄 수정하고, 판단이 필요한 경우 `remaining_conflicts`로 표시해 사람이 검토하게 한다.

### 재실행 기준

- Phase 4 (derive)를 재실행한 경우
- 새로운 enum 타입이 추가된 경우
- 기능 요구사항 문서 또는 UI 요구사항 문서를 재생성한 경우
- 상태값 변환 모듈이 수정된 경우

| 항목 | 결정 | 이유 |
|---|---|---|
| 슬라이스 단위 | 원자적 인과 연쇄 (트랜잭션 경계 기준) | 진입점 단위로 묶으면 Rule Record가 과도하게 커짐 |
| 교차검증 범위 | group + derive만 | 추론이 일어나는 단계에만 적용, extract/audit은 결정론적 |
| Call Graph 역할 | 슬라이스 backward traversal 인프라 | 요구사항 출력물이 아니라 탐색 도구 |
| 에러 코드 형식 | 4-tuple (condition, name, message, http_status) | 문자열만 저장하면 실패 경로의 맥락이 손실됨 |
| 클라이언트 분석 위치 | Phase 1 (extract), Phase 2 enrichment + CL Rule Record, Phase 5 FR 생성 | 사실 추출은 Phase 1, 소비는 각 단계별로 |
| 클라이언트 컴포넌트 슬라이싱 방식 | LLM이 컴포넌트 파일을 직접 읽어 판단 (정규식 패턴 매칭 배제) | 정규식은 프레임워크·컨벤션에 종속되어 다른 프로젝트에서 재사용 불가 |
| `string` 파라미터 실제 의미 판단 | 정적 추출(validation 메시지 + API 파라미터명)을 단서로 AI가 연결 판단 | 백엔드 타입 `string`에 UI 선택값을 인코딩하는 패턴은 타입만으로 감지 불가. 단, AI는 발견이 아닌 연결만 수행해야 hallucination 억제 |
| 클라이언트 용어 앵커링 | API URL → EP-ID → FR-ID 체인 | 독립 라벨링 시 백엔드와 용어 drift 발생 |
| 클라이언트 분석 수준 | Level 1 (100% 정적) + Level 2 (구조 검출 후 AI) | Level 3 (순수 AI) 배제로 hallucination 최소화 |
| 스키마 추출 위치 | Phase 2 후처리 | Rule Record 생성 직후 struct 레지스트리와 연결 |
| 스키마 전파 위치 | Phase 4 후처리 | FR 생성 직후 evidence_refs 체인으로 전파 |
| UI 커버리지 검증 | Phase 5 audit | 백엔드 커버리지와 함께 양방향 역검증 |
| 클라이언트 라우터 기반 도메인 분리 | Phase 3 group | 백엔드 모듈이 하나여도 클라이언트 라우트 첫 세그먼트가 다르면 별도 사용자 도메인 |
| 상태 코드 표시 텍스트 추출 | Phase 1 extract (Level 1) | 코드 상수와 UI 표시 텍스트는 다른 사실 — 둘 다 포착해야 재구현 가능 |
| UI 행동 → FR ui 섹션 통합 | Phase 4 derive | CL-*로 분리하면 백엔드 FR과 프론트 동작이 따로 놀아 재구현 시 참조 분산 |
| display_fields 명시 | Phase 4 derive | response_schema에 필드가 있어도 "이 필드가 화면에 표시된다"는 정보는 별도 |
| 정적 BFS unreachable ≠ dead code | Phase 5 audit | interface dispatch 등 동적 호출은 정적 BFS로 추적 불가 — 구현 레이어는 경로 기반으로 utility 분류 |
| 필드 생애주기 분석 위치 | Phase 4 (group 이후, derive 이전) | EP별 독립 분석(slice)만으로는 교차 EP 비교가 불가 — 도메인 그룹이 확정된 뒤에야 같은 엔티티의 여러 EP를 나란히 놓을 수 있음 |
| 불변 필드 탐지 방법 | UI disabled + 스키마 교차 참조 | 백엔드 코드에는 "이 필드를 변경하지 않는다"는 명시적 guard가 없는 경우가 대부분 — UI의 disabled 상태가 유일한 구조적 근거 |
| 편집 폼 disabled 필터링 정책 | `disabled="true"` 리터럴만 불변 제약으로 분류 | 비즈니스 조건부 disabled(`status === 'X'`)는 `client_behaviors.json`의 disabled_conditions로 처리 — 두 패턴을 혼동하면 false alarm 발생 |
| 필드 명세의 derive 연동 | derived_requirement → FR.behavior.preconditions 자동 삽입 | derive 단계 LLM이 "과목코드 변경 불가"를 hallucination 없이 생성하려면 코드 사실에서 나온 필드 제약이 입력으로 주어져야 함 |
| enum 일관성 검증 위치 | Phase 7 reconcile (audit 이후) | audit은 커버리지, reconcile은 값의 정확성을 검증 — 역할이 다르므로 분리 |
| enum ground truth 소스 | 백엔드 타입 정의 파일 (코드) | LLM 추론값이 아닌 코드에서 직접 파싱 — 존재하지 않는 값과 코드 정의 값의 차이를 자동 탐지 |
| 요구사항 문서 2종 분리 | 기능 요구사항(LLM 도출) + 화면 구성 요구사항(정적 추출) | 신뢰도가 다른 두 문서를 분리 — 재구현 시 "무엇을"과 "어떻게 보이는지"를 독립적으로 참조 가능 |
| 화면 UI 구조 추출 | Phase 1 extract (Level 1) | 페이지 제목·테이블 컬럼·폼 필드는 template의 구조적 사실 — hallucination 없이 정적으로 추출 가능 |
| 자동화 수준 | 완전 자동 | 단계별 파일 저장으로 중간 재실행 보장 |
